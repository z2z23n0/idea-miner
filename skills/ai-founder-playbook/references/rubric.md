# Rubric

评分只用于暴露判断结构，不要把总分当成机械答案。每项 0-4 分：0 = 没证据或明显失败，2 = 有早期信号，4 = 强证据且可重复。

## 目标类型分流

- **商业产品**：目标是收入、利润、融资、可持续公司。重点看买方、预算、付费意愿、CAC、毛利、销售周期和留存。
- **开源影响力项目**：目标是用户、GitHub stars、安装量、fork、贡献者、生态引用、口碑和开发者心智。重点看开发者痛点、README 可懂性、安装摩擦、demo 时刻、issue 活跃、维护负担和传播渠道。
- **混合型**：先用开源获取信任和分发，再通过 hosted/cloud、enterprise、support、插件市场、usage-based API 或服务变现。必须同时验证开源 adoption 和商业转化路径。

## 12 项判断维度

1. **痛点强度**：问题是否高频、高代价、必须解决。
2. **ICP 精确度**：是否能具体到职位、公司类型、工作流、触达场所。
3. **现有替代痛苦**：当前方案是否昂贵、慢、错、合规风险高，或需要大量人力。
4. **买方和预算 / 开源受众**：商业产品看谁付钱、付多少、采购路径；开源项目看谁会安装、star、fork、贡献、在真实项目中依赖。
5. **时机**：监管、技术、成本、组织行为或市场结构是否形成顺风。
6. **AI 原生杠杆**：AI 是否改变交付方式、成本结构、速度或质量，而不只是套壳。
7. **可验证输出**：AI 结果能否被检查、回滚、审计，并有合理人工兜底。
8. **分发楔子**：创始人是否有可信、低成本、可重复的触达路径。
9. **早期证据**：访谈、留存、付费、LOI、试点、复购、推荐；开源项目则看 stars/forks/downloads、issue、PR、Discord/社区、教程、真实依赖。
10. **单位经济 / 维护经济**：商业产品看推理、人工复核、集成、支持、销售成本；开源项目看维护成本、兼容性负担、issue/support 压力和可持续投入。
11. **防守性**：数据复利、工作流嵌入、品牌信任、合规资质、网络效应或渠道壁垒。
12. **创始人优势**：领域知识、渠道、速度、品味、信任、执行力是否和机会匹配。

## 判读区间

- **36-48**：可以推进，但仍要列明最危险假设和下一轮验证。
- **26-35**：有苗头，先做一轮聚焦验证，不要急着扩功能或扩市场。
- **16-25**：需要重构假设；通常是 ICP、痛点、买方或分发没定。
- **0-15**：不建议投入构建，除非出现新的强证据。

总分高但任一关键项为 0，要降级处理。商业产品的一票否决项：没有买方、没有痛点、AI 成本无法覆盖、合规风险不可承受、无法触达目标用户。开源项目的一票否决项：没有清晰开发者受众、安装后没有 aha moment、已有强开源替代且无差异化、维护成本明显失控、没有可信传播渠道。

## 实验菜单

按阶段挑最小实验，不要一次塞满。

- **问题访谈 sprint**：只有当访谈是当前最短证据路径时才做；找精确 ICP，重点问过去行为、当前流程、代价和替代方案，不机械追求固定人数。
- **竞品客户抱怨综合**：抓取公开评论、社区帖子、论坛和案例，找重复抱怨与未满足需求。
- **Concierge / Wizard-of-Oz**：用人工和现成工具模拟核心价值，看用户是否愿意反复用或付费。
- **Fake-door / 预售页**：测试细分市场对具体价值主张的点击、预约、付费意向。
- **付费试点**：明确交付结果、时间、价格、成功标准和退出条件。
- **留存队列**：按 cohort 看第 1/2/4/8 周是否继续完成核心任务。
- **销售摩擦测试**：逐步提高价格，观察客户从“立刻买”到“需要想想”的临界点。
- **创始人离线测试**：假设创始人一周不在，找出会停摆的流程和决策。
- **开源 repo fake-door**：写 README、demo gif、API 示例和 issue 模板，发到目标社区，看 star、watch、issue、fork、试用反馈。
- **开发者最小 demo**：只做能在 5 分钟内跑起来的核心 demo，验证安装摩擦和 aha moment。
- **开源分发测试**：在 HN Show HN、Reddit、GitHub topic、X thread、相关 Discord/Slack 发一个真实 demo，观察收藏和真实使用的比例。

## AI 原生专项检查

- **Copilot、Agent、AI-enabled service 分清楚**：辅助人、自动执行工作流、还是 AI+人工交付服务，会决定定价、责任和成本。
- **收费指标贴近价值**：技术买家可理解用量；业务买家更理解任务、工作流或结果。
- **优先考虑混合定价**：基础费用覆盖固定成本，使用量/结果层捕获增量价值。
- **不要用成本加成作为主要叙事**：客户买的是 ROI、风险下降或业务结果，不是模型调用。
- **数据上下文是产品的一部分**：企业 AI 不只看模型，还看数据新鲜度、定义一致性、权限和语义层。
- **为代理可读性设计**：如果产品会被 AI agent 使用，API、事件、文档、权限和状态要机器可理解。
- **护城河来自复利循环**：用户行为、人工修正、偏好、工作流模板、集成和测试集要持续反哺产品。

## 开源项目专项检查

- **Star 不是唯一指标**：stars 是注意力，安装量、issue、fork、PR、真实依赖和二次传播才更接近使用。
- **README 是产品界面**：一句话定位、安装命令、30 秒 demo、对比表和常见用例决定转化。
- **生态位要窄**：开源项目更需要占一个清晰关键词和 GitHub topic，不要一开始做“平台”。
- **维护成本要前置**：兼容性、用户支持、安全漏洞、breaking changes、roadmap 都是成本。
- **商业化要诚实**：如果未来要变现，尽早明确 hosted/cloud、enterprise、support、插件、license 或 API 路径，避免社区信任反噬。

## 常用追问

- 最近一次用户真实遇到这个问题是什么时候？他们当时花了多少钱、多少时间、承担了什么风险？
- 如果这个产品明天消失，用户会回到什么替代方案？会有多痛？
- 买方为什么今年一定要解决，而不是继续忍一年？
- 你最容易接触的前 20 个目标用户是谁？为什么他们会愿意跟你聊？
- 哪个指标会证明这不是好奇心流量，而是真需求？
- 哪个竞品只要做一个 feature 就会让你很难活？
- 这个业务在 10 个客户时单位经济能否成立？如果不能，为什么 1000 个客户会变好？
- 哪些任务必须由 AI 自动完成，哪些必须让人确认？
- 如果这是开源项目，为什么开发者会 star 之后真的安装？为什么会在自己的项目里依赖？
- 这个 repo 的 30 秒 demo 是什么？用户第一次 aha moment 在哪里？
- 你要的是 star、安装量、贡献者、生态引用、还是后续商业化？这些目标会互相冲突吗？
